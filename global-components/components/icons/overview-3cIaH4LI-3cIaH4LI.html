Icons

Icons serve as a visual language, providing a consistent and recognizable set of symbols to communicate actions, concepts, or status across an interface.

Brand expression

Tripadvisor icons are designed to be simple, modern, and accessible. Icons were a focal point of Tripadvisorâ€™s 2020 brand redesign, and the icon set was redesigned and expanded to be lighter and more fun. Each icon is reduced to its minimal form (icons are typically stroked instead of filled) and only express essential characteristics. The icon set is crafted to represent common actions and items.

Design usage guidelines

When to use icons

Product design guidelines say to â€œUse icons with restraintâ€. Our design is clean, minimal, and clutter-free; icons should not distract users from their goals. Icons must not be overused but implemented intentionally (when in doubt, keep it out!). Icons can help ease usersâ€™ cognitive load by quickly relaying or enforcing a message, but keep in mind that some icon meanings are difficult to interpret for users unfamiliar with our brand or may have different meanings in different cultures. Use icons to highlight a specific message or action and when they clearly communicate a specific meaning to users that will help them to better understand the message or action.


In most cases, it is a best practice to use an icon to complement a text label or message, as it is not accessible for most users to depend on the icon alone to relay a message.

Icon names and categories

Each icon is named to dictate its usages; however, some icons have usages that can extend beyond the icon names. Refer to the iconâ€™s metadata for all icon usages: You can find the icon metadata in Figma by hovering over the icon in the asset panel or in the design and inspect panel (images ordered left to right below). Metadata will also be surfaced when you search for an icon in the assets panel. If you want to use the icon for a usage that is not listed in the metadata or the name, please review with the design team first- if this usage is approved, alert the DS team to update the metadata of the icon (at the moment icon metadata is only used for designers and Figma - if metadata changes engineering does not need to be alerted).

Stroked vs. Filled

Designers should default to use the stroked version of icons and use the filled version thoughtfully and intentionally. Noteworthy exception: Some icons have state changes that use the filled icon version, for example when a user â€œlikesâ€ something the heart changes from the stroked to the filled version.

 

 

 

Size

 

Icon component

For product design, we provide the following icon sizes. Icon sizes are named semantically. They start at icon-01 and increase in ascending order both in the name and unit size. The standard size for these icons is icon-03, which is 24 x 24px.

 

Components with icon

If a component has an icon inside of it, the component size is pre-set and the size of the icon cannot be adjusted.

Clickable components typically use size icon-02 (size 20px) for decorative icons, dropdown chevrons are typically icon-01 (16px).

 

 

 

Colors

Icons typically inherit the color of the surrounding text. For example, if an icon is inline with a line of text, the icon will be the same color as that line of text (e.g. onSurface). If the icon is inside of a button, the icon will be the same color as the button text (e.g. onPrimary) and will also inherit any state changes in colors (this is subject to change when button 2.0 is released).

Accessibility

Dynamic Type

On Native, icons are often treated like text and will scale with dynamic type. As a general guideline, decorative icons should be unresponsive to dynamic type and should not scale to different sizes because they are not critical to the userâ€™s goals and may instead disrupt the user flow at larger sizes. If the icon is functional, for example an X icon to close an element, the designers may consider whether or not the icon should respond to dynamic type (these scenarios are contextual and should be considered on a case by case basis in the component documentation). Designers should always consider and annotate whether or not icons should scale with dynamic type or ignore it.
