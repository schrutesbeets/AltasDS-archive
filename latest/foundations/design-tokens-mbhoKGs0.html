Design tokens

Design tokens are a single source of truth for all design-related decisions, providing a way to codify and manage the visual properties of a design system.

 

 

 

Overview

 

Design tokens are essentially shortcuts to specific style values like colors, fonts, and measurements. Instead of using hardcoded values directly in your design system, you'll use these tokens.

Each token is clearly named to indicate its purpose and where it's applied (for instance, --atlas-scheme-surface designates the container color for a page or component surface). This means that even if the actual value a token points to changes, its name and how it's used stay consistent.

The Atlas Design System organizes these tokens into three main categories: core, system, and component.

 

 

 

Core tokens

These tokens make up all the style options available in a design system. They usually point to a static value—like a color hex code or font size—but can also point to other core tokens. Core tokens don't change based on context.

System tokens

System tokens are crucial for establishing a consistent design language within a specific theme or context, acting as the bridge between a design system and its intention in the user interface. They define the intended purpose of a core token within the UI and are the primary mechanism for theming, allowing a single system token to adapt to different contexts, such as light or dark modes, by pointing to various reference tokens. To maintain flexibility and scalability, system tokens should always link to reference tokens rather than static values.

Component tokens

Component tokens are designed to represent the individual elements that make up a component, encompassing aspects like containers, labels, icons, and their various states, along with their associated values such as size, shape, color, or elevation. These tokens should ideally point to system or core tokens rather than containing hardcoded values, ensuring greater flexibility and consistency. While not every stylistic detail of a component can be tokenized, tokens should be utilized whenever a design choice applies across multiple components with similar intentions.
