Empty State

Empty states appear when expected content isn’t available or cannot be displayed. This can include first-time use of a feature, after completing or clearing tasks, when a search yields no results, or when an error prevents content from loading.

  • Overview
  • Dos & Don'ts

Properties

 

Leading asset

Leading assets in empty states are optional and may include illustrations. Any illustration should be contextually relevant, reinforcing the message of the empty or error state rather than distracting from it.

 

Content

The title should be a few words to state why the screen is empty, clearly explain the error, or what action to take. Body text should support it and clarify the next step and what content will appear.

 

CTA

Empty states should not feel like dead ends, so be sure to include a CTA when it makes sense. If not, provide a link or clear guidance so users know what to do next.

Inline vs. Full screen

Use an inline empty patterns when the error appears within a screen that has a title at the top. For a standalone empty states, use the full-screen template with the action button docked at the bottom.

 

Inline pattern

Used when placed within another screen that has a title at the top. The action button is positioned close to the content, keeping it contextually tied and easily discoverable.

Full screen pattern

Used to indicate that something has occurred during a flow or process, often as a standalone screen. The action button is docked or sticky at the bottom, ensuring it’s easily accessible for the user’s next step.

Title

Do

Don't

Do not utilize punctuation within state titles.

 

Primary action

Do

Don't

Do not have primary actions be a dead end for users.
